# 任务：[FEATURE NAME]

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档  
**前置条件**：plan.md（必填）、spec.md（用户故事必填）、research.md、data-model.md、contracts/

**测试**：下列示例包含测试任务。测试为可选项——仅在功能规格中明确要求时纳入。

**组织方式**：任务按用户故事分组，以便各故事可独立实现与测试。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可并行（不同文件、无依赖）
- **[Story]**：所属用户故事（如 US1、US2、US3）
- 描述中须含确切文件路径

## 路径约定

- **单仓库**：根目录下 `src/`、`tests/`
- **Web 应用**：`backend/src/`、`frontend/src/`
- **移动端**：`api/src/`、`ios/src/` 或 `android/src/`
- 下列路径按单项目假设——请根据 plan.md 结构调整

## 依赖与执行顺序

### 阶段依赖

- **搭建（阶段 1）**：无依赖——可立即开始
- **基础（阶段 2）**：依赖搭建完成——阻塞所有用户故事
- **用户故事（阶段 3 及以后）**：均依赖基础阶段完成
  - 之后用户故事可并行（若人力允许）
  - 或按优先级顺序（P1 → P2 → P3）
- **打磨（最后阶段）**：依赖所需用户故事均完成

### 用户故事依赖

- **用户故事 1（P1）**：基础（阶段 2）完成后可开始——不依赖其他故事
- **用户故事 2（P2）**：基础（阶段 2）完成后可开始——可与 US1 集成但应可独立测试
- **用户故事 3（P3）**：基础（阶段 2）完成后可开始——可与 US1/US2 集成但应可独立测试

### 各用户故事内部

- 测试（若包含）必须先写并在实现前失败
- 模型先于服务
- 服务先于端点
- 核心实现先于集成
- 完成当前故事后再进入下一优先级

### 并行机会

- 阶段 1 中标记 [P] 的搭建任务可并行
- 阶段 2 中标记 [P] 的基础任务可并行（在阶段 2 内）
- 基础阶段完成后，各用户故事可并行开始（若团队容量允许）
- 某故事中标记 [P] 的测试可并行
- 故事中标记 [P] 的模型可并行
- 不同用户故事可由不同成员并行推进

## 实现策略

### MVP 优先（仅用户故事 1）

1. 完成阶段 1：搭建
2. 完成阶段 2：基础（关键——阻塞所有故事）
3. 完成阶段 3：用户故事 1
4. **暂停并验证**：独立测试用户故事 1
5. 就绪则部署/演示

### 增量交付

1. 完成搭建 + 基础 → 基础就绪
2. 增加用户故事 1 → 独立测试 → 部署/演示（MVP！）
3. 增加用户故事 2 → 独立测试 → 部署/演示
4. 增加用户故事 3 → 独立测试 → 部署/演示
5. 每个故事在不破坏既有故事的前提下增加价值

### 多人文档并行策略

多人开发时：

1. 团队共同完成搭建 + 基础
2. 基础完成后：
   - 开发者 A：用户故事 1
   - 开发者 B：用户故事 2
   - 开发者 C：用户故事 3
3. 各故事独立完成并集成

## 说明

- [P] 任务 = 不同文件、无依赖
- [Story] 标签将任务映射到具体用户故事以便追溯
- 每个用户故事应可独立完成与测试
- 实现前确认测试失败
- 每个任务或逻辑分组后提交
- 可在任检查点暂停以独立验证故事
- 避免：模糊任务、同文件冲突、破坏独立性的跨故事依赖
