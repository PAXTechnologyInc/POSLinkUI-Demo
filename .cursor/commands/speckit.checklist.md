## 检查清单定位：「需求的单元测试」

**核心概念**：检查清单是**需求文档写作的单元测试**——用于校验某一领域内需求的**质量、清晰度与完整性**。

**不用于**验证/测试实现：

- ❌ 不是「验证按钮点击是否正确」
- ❌ 不是「测试错误处理是否工作」
- ❌ 不是「确认 API 是否返回 200」
- ❌ 不是检查代码/实现是否与规格一致

**用于**需求质量校验：

- ✅ 「是否已为所有卡片类型定义视觉层级需求？」（完整性）
- ✅ 「『突出展示』是否用具体尺寸/位置量化？」（清晰度）
- ✅ 「所有可交互元素的悬停态需求是否一致？」（一致性）
- ✅ 「是否为键盘导航定义无障碍需求？」（覆盖度）
- ✅ 「规格是否规定 Logo 图片加载失败时的行为？」（边界情况）

**类比**：若规格是用自然语言写的「代码」，检查清单就是其单元测试套件。你检验的是需求是否**写得好、完整、无歧义、可落地**——而非实现是否**跑得对**。

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（若非空）。

## 执行步骤

1. **搭建**：在仓库根目录运行 `.specify/scripts/bash/check-prerequisites.sh --json`，解析 JSON 得到 FEATURE_DIR 与 AVAILABLE_DOCS。
   - 所有文件路径须为绝对路径。
   - 参数中含单引号（如 "I'm Groot"）时使用转义：例如 `'I'\''m Groot'`（或尽量用双引号：`"I'm Groot"`）。

2. **澄清意图（动态）**：生成最多**三个**初始情境澄清问题（无预制题库）。必须满足：
   - 根据用户措辞 + 从 spec/plan/tasks 提取的信号生成
   - 仅询问会**实质改变**检查清单内容的信息
   - 若 `$ARGUMENTS` 中已无疑义则**单独跳过**该问题
   - 宁可**精准**也不要泛泛

   生成算法：
   1. 提取信号：功能域关键词（如 auth、latency、UX、API）、风险信号（「关键」「必须」「合规」）、干系人提示（「QA」「评审」「安全团队」）、明确交付物（「a11y」「rollback」「contracts」）。
   2. 将信号聚类为候选关注域（最多 4 个）并按相关性排序。
   3. 若未明示，推断可能的受众与时机（作者、评审者、QA、发布）。
   4. 检测缺失维度：范围广度、深度/严格度、风险侧重、排除边界、可度量验收标准。
   5. 从下列原型中选问题：
      - 范围细化（例如：「是否应包含与 X、Y 的集成触点，还是仅限本地模块正确性？」）
      - 风险排序（例如：「下列潜在风险域中哪些必须做强制门禁检查？」）
      - 深度校准（例如：「这是轻量提交前 sanity 列表还是正式发布门禁？」）
      - 受众定位（例如：「仅供作者自用还是 PR 评审时由同事使用？」）
      - 边界排除（例如：「本轮是否明确排除性能调优类条目？」）
      - 场景类缺口（例如：「未检测到恢复流——回滚/部分失败路径是否在范围内？」）

   问题格式规则：
   - 若呈现选项，生成紧凑表格，列：选项 | 候选 | 为何重要
   - 选项最多 A–E；若自由作答更清晰则省略表格
   - 勿让用户重复已说内容
   - 避免臆测类别（勿幻觉）。若不确定，明确问：「请确认 X 是否属于范围。」

   无法交互时的默认：
   - 深度：标准
   - 受众：与代码相关则为评审者（PR）；否则为作者
   - 关注：相关性最高的前 2 个聚类

   输出问题（标 Q1/Q2/Q3）。回答后：若仍有 ≥2 个场景类（备选/异常/恢复/非功能域）不清晰，可再提最多**两个**针对性追问（Q4/Q5），每条附一行理由（例如：「恢复路径风险未决」）。总问题不超过五个。若用户明确拒绝更多追问则不再升级。

3. **理解用户请求**：合并 `$ARGUMENTS` 与澄清答案：
   - 推导检查清单主题（如安全、评审、部署、UX）
   - 合并用户明确提出的必含项
   - 将关注选择映射到类别骨架
   - 从 spec/plan/tasks 推断缺失上下文（**勿幻觉**）

4. **加载功能上下文**：自 FEATURE_DIR 读取：
   - spec.md：功能需求与范围
   - plan.md（若存在）：技术细节、依赖
   - tasks.md（若存在）：实现任务

   **上下文加载策略**：
   - 仅加载与当前关注域相关的必要片段（避免全文 dump）
   - 长节优先概括为简洁的场景/需求要点
   - 渐进披露：仅当发现缺口时再检索
   - 若源文档很大，先生成中间摘要项而非嵌入大段原文

5. **生成检查清单** — 创建「需求的单元测试」：
   - 若不存在则创建 `FEATURE_DIR/checklists/` 目录
   - 生成唯一检查清单文件名：
     - 按领域用短描述名（如 `ux.md`、`api.md`、`security.md`）
     - 格式：`[domain].md`
   - 文件处理：
     - 若文件**不存在**：新建文件，条目从 CHK001 起编号
     - 若文件**已存在**：向现有文件**追加**新项，从最后 CHK ID 继续（例如末条为 CHK015，则新项从 CHK016 起）
   - **切勿**删除或替换已有检查清单内容——始终保留并追加

   **核心原则 — 测需求，不测实现**：
   每条检查清单项必须评估**需求本身**是否：
   - **完整**：必要需求是否齐全？
   - **清晰**：是否无歧义且具体？
   - **一致**：需求之间是否对齐？
   - **可度量**：能否客观验证？
   - **覆盖**：场景/边界是否覆盖？

   **类别结构** — 按需求质量维度分组：
   - **需求完整性**（必要需求是否均已文档化？）
   - **需求清晰度**（是否具体、无歧义？）
   - **需求一致性**（是否无冲突？）
   - **验收标准质量**（成功标准是否可度量？）
   - **场景覆盖**（流程/用例是否覆盖？）
   - **边界覆盖**（边界条件是否定义？）
   - **非功能需求**（性能、安全、无障碍等是否写明？）
   - **依赖与假设**（是否记录并校验？）
   - **歧义与冲突**（哪些待澄清？）

   **如何撰写检查项 — 「需求的单元测试」**：

   ❌ **错误**（测实现）：
   - 「验证落地页展示 3 个剧集卡片」
   - 「测试桌面端悬停态是否工作」
   - 「确认点击 Logo 是否返回首页」

   ✅ **正确**（测需求质量）：
   - 「精选剧集的**数量与版式**是否在需求中明确规定？」[完整性]
   - 「『突出展示』是否用具体尺寸/位置量化？」[清晰度]
   - 「各可交互元素的悬停态需求是否一致？」[一致性]
   - 「是否**所有**可交互 UI 都定义了键盘导航需求？」[覆盖]
   - 「Logo 图片加载失败时的降级行为是否在规格中规定？」[边界]
   - 「异步剧集数据的加载态是否在需求中定义？」[完整性]
   - 「规格是否定义竞争 UI 元素之间的视觉层级？」[清晰度]

   **条目结构**：
   每条应遵循：
   - 以问句形式询问需求质量
   - 聚焦规格/计划中**写了什么**（或**没写什么**）
   - 在括号内标注质量维度 [Completeness/Clarity/Consistency/等]
   - 核对已有需求时引用规格节 `[Spec §X.Y]`
   - 核对缺失需求时使用标记 `[Gap]`

   **按质量维度的示例**：

   完整性：
   - 「是否已为所有 API 失败模式定义错误处理需求？[Gap]」
   - 「是否已为所有可交互元素规定无障碍需求？[Completeness]」
   - 「是否已定义响应式布局的移动端断点需求？[Gap]」

   清晰度：
   - 「『快速加载』是否用具体时延阈值量化？[Clarity, Spec §NFR-2]」
   - 「『相关剧集』的筛选准则是否明确？[Clarity, Spec §FR-5]」
   - 「『突出』是否用可度量视觉属性定义？[Ambiguity, Spec §FR-4]」

   一致性：
   - 「各页面的导航需求是否一致？[Consistency, Spec §FR-10]」
   - 「落地页与详情页的卡片组件需求是否一致？[Consistency]」

   覆盖：
   - 「是否为零内容状态（无剧集）定义需求？[Coverage, Edge Case]」
   - 「是否覆盖并发用户交互场景？[Coverage, Gap]」
   - 「是否规定部分数据加载失败时的需求？[Coverage, Exception Flow]」

   可度量性：
   - 「视觉层级需求是否可度量/可测试？[Acceptance Criteria, Spec §FR-1]」
   - 「『视觉权重平衡』能否客观验证？[Measurability, Spec §FR-2]」

   **场景分类与覆盖**（聚焦需求质量）：
   - 检查是否存在以下类型的需求：主路径、备选、异常/错误、恢复、非功能场景
   - 对每一类追问：「[某类] 场景的需求是否完整、清晰且一致？」
   - 若某类缺失：「[某类] 场景的需求是**有意排除**还是**遗漏**？[Gap]」
   - 若存在状态变更，纳入韧性/回滚：「迁移失败时是否定义回滚需求？[Gap]」

   **可追溯性要求**：
   - 至少 **≥80%** 的条目须含至少一处可追溯引用
   - 每条应引用：规格节 `[Spec §X.Y]`，或使用标记：`[Gap]`、`[Ambiguity]`、`[Conflict]`、`[Assumption]`
   - 若无 ID 体系：「是否已建立需求与验收标准的 ID 方案？[Traceability]」

   **暴露并解决问题**（需求质量问题）：
   就需求本身提问：
   - 歧义：「术语『快』是否用具体指标量化？[Ambiguity, Spec §NFR-1]」
   - 冲突：「§FR-10 与 §FR-10a 的导航需求是否冲突？[Conflict]」
   - 假设：「『播客 API 始终可用』的假设是否已验证？[Assumption]」
   - 依赖：「外部播客 API 的需求是否已文档化？[Dependency, Gap]」
   - 定义缺失：「『视觉层级』是否用可度量准则定义？[Gap]」

   **内容合并**：
   - 软上限：原始候选项 >40 时按风险/影响优先
   - 合并检查同一需求侧面的近似重复项
   - 若 >5 个低影响边界情况，可合并为一项：「需求是否覆盖边界情况 X、Y、Z？[Coverage]」

   **🚫 绝对禁止** — 以下会使其变成实现测试而非需求测试：
   - ❌ 以「Verify」「Test」「Confirm」「Check」+ 实现行为开头的条目
   - ❌ 引用代码执行、用户动作、系统运行时行为
   - ❌ 「正确显示」「正常工作」「按预期运行」
   - ❌ 「点击」「导航」「渲染」「加载」「执行」
   - ❌ 测试用例、测试计划、QA 流程
   - ❌ 实现细节（框架、API、算法）

   **✅ 必需模式** — 以下用于测需求质量：
   - ✅ 「[某类] 需求是否已为 [场景] 定义/规定/文档化？」
   - ✅ 「[模糊词] 是否用具体准则量化/澄清？」
   - ✅ 「[节 A] 与 [节 B] 的需求是否一致？」
   - ✅ 「[需求] 能否客观度量/验证？」
   - ✅ 「[边界/场景] 是否在需求中覆盖？」
   - ✅ 「规格是否定义了 [缺失方面]？」

6. **结构参考**：按 `.specify/templates/checklist-template.md` 的规范生成检查清单的标题、元信息节、类别标题与 ID 格式。若模板不可用，则使用：H1 标题、用途/创建元信息行、含 `- [ ] CHK### <需求项>` 的 `##` 类别节，CHK ID 从 CHK001 起全局递增。

7. **报告**：输出检查清单完整路径、条目数量，并说明本次是新建文件还是追加。摘要：
   - 所选关注域
   - 深度级别
   - 角色/时机
   - 用户明确提出的必含项是否已纳入

**说明**：每次 `/speckit.checklist` 调用使用简短、描述性文件名，或新建或追加。这样可：

- 支持多种类型检查清单（如 `ux.md`、`test.md`、`security.md`）
- 文件名易记且能体现用途
- 在 `checklists/` 文件夹中便于识别与导航

为避免杂乱，使用描述性类型，并在完成后清理过时检查清单。

## 检查清单类型示例与样例条目

**UX 需求质量：** `ux.md`

样例条目（测需求，**非**实现）：

- 「视觉层级需求是否用可度量准则定义？[Clarity, Spec §FR-1]」
- 「UI 元素数量与位置是否明确规定？[Completeness, Spec §FR-1]」
- 「交互状态（悬停、聚焦、激活）需求是否一致定义？[Consistency]」
- 「是否所有可交互元素都规定无障碍需求？[Coverage, Gap]」
- 「图片加载失败时的降级行为是否定义？[Edge Case, Gap]」
- 「『突出展示』能否客观度量？[Measurability, Spec §FR-4]」

**API 需求质量：** `api.md`

样例条目：

- 「是否所有失败场景都规定了错误响应格式？[Completeness]」
- 「限流需求是否用具体阈值量化？[Clarity]」
- 「各端点的认证需求是否一致？[Consistency]」
- 「是否对外部依赖定义重试/超时需求？[Coverage, Gap]」
- 「版本策略是否在需求中说明？[Gap]」

**性能需求质量：** `performance.md`

样例条目：

- 「性能需求是否用具体指标量化？[Clarity]」
- 「是否所有关键用户旅程都定义性能目标？[Coverage]」
- 「不同负载下的性能需求是否规定？[Completeness]」
- 「性能需求能否客观度量？[Measurability]」
- 「高负载场景下的降级需求是否定义？[Edge Case, Gap]」

**安全需求质量：** `security.md`

样例条目：

- 「是否所有受保护资源都规定认证需求？[Coverage]」
- 「敏感信息的数据保护需求是否定义？[Completeness]」
- 「威胁模型是否已文档化且需求与之对齐？[Traceability]」
- 「安全需求是否与合规义务一致？[Consistency]」
- 「安全故障/泄露响应需求是否定义？[Gap, Exception Flow]」

## 反例：不应做什么

**❌ 错误 — 这些在测实现，不是需求：**

```markdown
- [ ] CHK001 - 验证落地页是否展示 3 个剧集卡片 [Spec §FR-001]
- [ ] CHK002 - 测试桌面端悬停态是否正常工作 [Spec §FR-003]
- [ ] CHK003 - 确认点击 Logo 是否跳转首页 [Spec §FR-010]
- [ ] CHK004 - 检查相关剧集区是否显示 3–5 条 [Spec §FR-005]
```

**✅ 正确 — 这些在测需求质量：**

```markdown
- [ ] CHK001 - 精选剧集的数量与版式是否在需求中明确规定？[Completeness, Spec §FR-001]
- [ ] CHK002 - 各可交互元素的悬停态需求是否一致定义？[Consistency, Spec §FR-003]
- [ ] CHK003 - 所有可点击品牌元素的导航需求是否清晰？[Clarity, Spec §FR-010]
- [ ] CHK004 - 相关剧集的筛选准则是否已文档化？[Gap, Spec §FR-005]
- [ ] CHK005 - 异步剧集数据的加载态需求是否已定义？[Gap]
- [ ] CHK006 - 「视觉层级」需求能否客观度量？[Measurability, Spec §FR-001]
```

**关键差异：**

- 错：测系统是否工作正确
- 对：测需求是否写得正确
- 错：验证行为
- 对：验证需求质量
- 错：「它是否做了 X？」
- 对：「X 是否被说清楚？」
