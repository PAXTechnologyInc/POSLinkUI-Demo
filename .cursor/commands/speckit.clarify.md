## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（若非空）。

## 概要

**目标**：发现功能规格中的歧义或缺失决策点并降低其影响，将澄清直接记录在规格文件中。

**说明**：本澄清流程应在调用 `/speckit.plan`**之前**完成。若用户明确表示跳过澄清（例如探索性 spike），可继续，但须警告下游返工风险增加。

**执行步骤**：

1. 在仓库根目录**一次性**运行 `.specify/scripts/bash/check-prerequisites.sh --json --paths-only`（组合 `--json --paths-only` / `-Json -PathsOnly`）。解析最小 JSON 字段：
   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   - （可选捕获 `IMPL_PLAN`、`TASKS` 供后续链式流程。）
   - 若 JSON 解析失败，中止并指导用户重新运行 `/speckit.specify` 或检查功能分支环境。
   - 参数中含单引号（如 "I'm Groot"）时使用转义：例如 `'I'\''m Groot'`（或尽量用双引号：`"I'm Groot"`）。

2. 加载当前规格文件。按下述分类做结构化歧义与覆盖扫描。各类别标记状态：Clear / Partial / Missing。生成内部覆盖图用于排序（若无问题可不问，则勿输出原始图）。

   **功能范围与行为：**
   - 核心用户目标与成功标准
   - 明确的范围外声明
   - 用户角色 / 人设区分

   **领域与数据模型：**
   - 实体、属性、关系
   - 身份与唯一性规则
   - 生命周期/状态迁移
   - 数据量/规模假设

   **交互与 UX 流程：**
   - 关键用户旅程/序列
   - 错误/空态/加载态
   - 无障碍或本地化说明

   **非功能质量属性：**
   - 性能（延迟、吞吐目标）
   - 可扩展性（水平/垂直、上限）
   - 可靠性与可用性（在线时间、恢复预期）
   - 可观测性（日志、指标、追踪信号）
   - 安全与隐私（认证/授权、数据保护、威胁假设）
   - 合规/监管约束（若有）

   **集成与外部依赖：**
   - 外部服务/API 及失败模式
   - 数据导入/导出格式
   - 协议/版本假设

   **边界情况与失败处理：**
   - 负面场景
   - 限流/节流
   - 冲突解决（例如并发编辑）

   **约束与权衡：**
   - 技术约束（语言、存储、托管）
   - 明确权衡或已否决方案

   **术语与一致性：**
   - 规范术语表
   - 避免的同义词/弃用术语

   **完成信号：**
   - 验收标准可测试性
   - 可度量的「完成定义」类指标

   **杂项 / 占位：**
   - TODO 标记/未决决策
   - 缺乏量化的模糊形容词（「健壮」「直观」等）

   对状态为 Partial 或 Missing 的类别，除非以下情况，否则增加候选问题机会：
   - 澄清不会实质改变实现或验证策略
   - 信息更适合推迟到计划阶段（内部记录即可）

3. **在内部**生成候选澄清问题优先级队列（最多 5 个）。**不要**一次性全部输出。约束如下：
    - 整个会话最多 5 个问题。
    - 每个问题须可用以下**之一**回答：
       - 简短多选（2–5 个互斥选项），或
       - 一词/短短语（明确约束：「回答不超过 5 个词」）。
    - 仅纳入答案会实质影响架构、数据建模、任务拆分、测试设计、UX 行为、运维就绪或合规验证的问题。
    - 平衡类别覆盖：优先最高影响的未决类别；避免在单一高影响领域（如安全态势）未决时问两个低影响问题。
    - 排除已回答、琐碎风格偏好或计划层执行细节（除非阻塞正确性）。
    - 优先能减少下游返工或防止验收测试错位的澄清。
    - 若未决类别超过 5 个，按（影响 × 不确定性）启发式取前 5。

4. **顺序提问循环（交互）**：
    - **一次只展示一个问题**。
    - 对多选题：
       - **分析所有选项**并基于以下确定**最合适选项**：
          - 该类型项目的最佳实践
          - 类似实现中的常见模式
          - 风险降低（安全、性能、可维护性）
          - 与规格中可见的明确目标或约束的对齐
       - 在顶部**突出展示推荐选项**及简明理由（1–2 句说明为何最佳）。
       - 格式：`**推荐：** 选项 [X] - <理由>`
       - 然后将所有选项渲染为 Markdown 表格：

       | 选项 | 说明 |
       |------|------|
       | A | <选项 A 说明> |
       | B | <选项 B 说明> |
       | C | <选项 C 说明>（按需增至 D/E，最多 5 个） |
       | Short | 提供其他简短答案（≤5 词）（仅当自由作答合适时包含） |

       - 表格后追加：`可回复选项字母（如 "A"），或回复 "yes"/"recommended" 接受推荐，或给出你自己的简短答案。`
    - 对短答式（无有意义离散选项）：
       - 基于最佳实践与上下文给出**建议答案**。
       - 格式：`**建议：** <拟议答案> - <简要理由>`
       - 然后输出：`格式：简短答案（≤5 词）。可回复 "yes" 或 "suggested" 接受建议，或给出你自己的答案。`
    - 用户回答后：
       - 若用户回复 "yes"、"recommended" 或 "suggested"，采用你先前给出的推荐/建议作为答案。
       - 否则校验答案对应某一选项或符合「≤5 词」约束。
       - 若模糊，请求快速消歧（仍计为同一问题，不前进）。
       - 满意后记入工作记忆（尚未写盘），进入下一队列问题。
    - 在以下情况停止继续提问：
       - 关键歧义已提前全部解决（剩余队列项不再需要），或
       - 用户表示结束（"done"、"good"、"no more"），或
       - 已问满 5 个问题。
    - **切勿**提前透露后续队列问题。
    - 若开始即无有效问题，立即报告无关键歧义。

5. **每次接受答案后集成（增量更新）**：
    - 维护内存中的规格表示（开始时加载一次）及原始文件内容。
    - 本会话中**首次**集成答案时：
       - 确保存在 `## Clarifications` 节（若缺失，在规格模板中最高层背景/概述节之后创建）。
       - 其下为当日创建（若不存在）`### Session YYYY-MM-DD` 子标题。
    - 接受后立即追加一条要点：`- Q: <question> → A: <final answer>`。
    - 然后将澄清应用到最合适的一节或多节：
       - 功能歧义 → 在功能需求中更新或增加要点。
       - 用户交互/角色区分 → 更新用户故事或参与者小节（若有）中的角色、约束或场景。
       - 数据形态/实体 → 更新数据模型（字段、类型、关系），保持顺序；简洁注明新增约束。
       - 非功能约束 → 在非功能/质量属性节增加或可度量指标（将模糊形容词转为指标或明确目标）。
       - 边界/负面流程 → 在边界情况/错误处理下新增要点（或若模板有占位则创建该小节）。
       - 术语冲突 → 在全文统一术语；必要时保留一次 `(formerly referred to as "X")`。
    - 若澄清使先前模糊陈述失效，**替换**该陈述而非重复；不留过时矛盾文本。
    - **每次**集成后保存规格文件以降低上下文丢失风险（原子覆盖写入）。
    - 保持格式：勿重排无关小节；保持标题层级。
    - 每条插入的澄清保持最小且可测试（避免叙事发散）。

6. **校验**（每次写入后及最后总检）：
   - Clarifications 会话中每个已接受答案恰好一条要点（无重复）。
   - 已问（已接受）问题总数 ≤ 5。
   - 更新后的节中不留本答案应已消除的模糊占位。
   - 无不矛盾的旧陈述（扫描并移除已无效的可选方案）。
   - Markdown 结构有效；仅允许新增标题：`## Clarifications`、`### Session YYYY-MM-DD`。
   - 术语一致：更新节中使用同一规范术语。

7. 将更新后的规格写回 `FEATURE_SPEC`。

8. **完成报告**（提问循环结束或提前终止后）：
   - 已问已答问题数量。
   - 更新后规格路径。
   - 触及的小节（列出名称）。
   - 覆盖摘要表：各 taxonomy 类别状态为 Resolved（原 Partial/Missing 已处理）、Deferred（超出配额或更适合计划）、Clear（已充分）、Outstanding（仍 Partial/Missing 但影响低）。
   - 若存在 Outstanding 或 Deferred，建议是否继续 `/speckit.plan` 或在计划后再次运行 `/speckit.clarify`。
   - 建议的下一命令。

**行为规则**：

- 若无实质歧义（或所有潜在问题影响都低），回复：「未检测到值得正式澄清的关键歧义。」并建议继续。
- 若规格文件缺失，指导用户先运行 `/speckit.specify`（勿在此新建规格）。
- 已问问题总数不得超过 5（单题澄清重试不计为新问题）。
- 除非缺失会阻塞功能清晰性，否则避免臆测技术栈问题。
- 尊重用户提前终止（"stop"、"done"、"proceed"）。
- 若因覆盖充分未提问，输出紧凑覆盖摘要（各类别均为 Clear）后建议推进。
- 若配额用尽仍有高影响未决类别，须在 Deferred 下明确标注并说明理由。
