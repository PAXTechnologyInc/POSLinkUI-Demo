## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（若非空）。

## 目标

在实现前识别三份核心产物（`spec.md`、`plan.md`、`tasks.md`）之间的不一致、重复、歧义与欠规定项。本命令**仅**在 `/speckit.tasks` 已成功生成完整 `tasks.md` 之后运行。

## 运行约束

**严格只读**：**不得**修改任何文件。输出结构化分析报告。可提供可选的修复方案（用户须明确批准后才可手动执行后续编辑类命令）。

**宪章权威**：项目宪章（`.specify/memory/constitution.md`）在本分析范围内**不可协商**。与宪章冲突自动为 **CRITICAL**，须调整 spec、plan 或 tasks——不得弱化、重新解释或默默忽略原则。若原则本身需变更，须在 `/speckit.analyze` 之外的单独、显式宪章更新中进行。

## 执行步骤

### 1. 初始化分析上下文

在仓库根目录运行一次 `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks`，解析 JSON 得到 FEATURE_DIR 与 AVAILABLE_DOCS。推导绝对路径：

- SPEC = FEATURE_DIR/spec.md
- PLAN = FEATURE_DIR/plan.md
- TASKS = FEATURE_DIR/tasks.md

若任一必需文件缺失则中止并提示错误（指导用户运行缺失的前置命令）。  
参数中含单引号（如 "I'm Groot"）时使用转义：例如 `'I'\''m Groot'`（或尽量用双引号：`"I'm Groot"`）。

### 2. 加载产物（渐进披露）

仅从各产物加载最少必要上下文：

**自 spec.md：**

- 概述/背景
- 功能需求
- 非功能需求
- 用户故事
- 边界情况（若有）

**自 plan.md：**

- 架构/技术栈选择
- 数据模型引用
- 阶段
- 技术约束

**自 tasks.md：**

- 任务 ID
- 描述
- 阶段分组
- 并行标记 [P]
- 引用的文件路径

**自宪章：**

- 加载 `.specify/memory/constitution.md` 用于原则校验

### 3. 构建语义模型

建立内部表示（输出中勿包含原始全文）：

- **需求清单**：每条功能与非功能需求，带稳定键（由祈使短语派生 slug；例如 "User can upload file" → `user-can-upload-file`）
- **用户故事/行为清单**：离散用户行为与验收标准
- **任务覆盖映射**：将各任务映射到一条或多条需求或故事（通过关键词 / 显式引用如 ID 或关键短语推断）
- **宪章规则集**：提取原则名称与 MUST/SHOULD 规范性陈述

### 4. 检测轮次（省 token 分析）

聚焦高信号发现。总计不超过 50 条；其余在溢出摘要中聚合。

#### A. 重复检测

- 识别近似重复的需求
- 标记可合并的较差表述

#### B. 歧义检测

- 标记缺少可度量标准的模糊形容词（fast、scalable、secure、intuitive、robust 等）
- 标记未解决占位符（TODO、TKTK、???、`<placeholder>` 等）

#### C. 欠规定

- 有动词但缺少对象或可度量结果的需求
- 用户故事与验收标准不对齐
- 任务引用 spec/plan 中未定义的文件或组件

#### D. 宪章对齐

- 任何与 MUST 原则冲突的需求或计划条目
- 宪章要求的章节或质量门禁缺失

#### E. 覆盖缺口

- 无任何关联任务的需求
- 无映射需求/故事的任务
- 未在任务中体现的非功能需求（如性能、安全）

#### F. 不一致

- 术语漂移（同一概念在不同文件中名称不同）
- plan 引用但 spec 中无的实体（或相反）
- 任务顺序矛盾（例如在无依赖说明时，集成任务早于基础搭建）
- 冲突需求（例如一处要求 Next.js，另一处指定 Vue）

### 5. 严重级别赋值

使用下列启发式排序：

- **CRITICAL**：违反宪章 MUST、缺失核心 spec 产物、或零覆盖且阻塞基线功能的需求
- **HIGH**：重复或冲突需求、安全/性能属性歧义、不可测试的验收标准
- **MEDIUM**：术语漂移、非功能任务覆盖缺失、边界情况欠规定
- **LOW**：文风/措辞改进、不影响执行顺序的轻微冗余

### 6. 输出精简分析报告

输出 Markdown 报告（不写文件），结构如下：

## 规格分析报告

| ID | 类别 | 严重级别 | 位置 | 摘要 | 建议 |
|----|------|----------|------|------|------|
| A1 | Duplication | HIGH | spec.md:L120-134 | 两条相似需求 … | 合并表述，保留更清晰版本 |

（每条发现一行；生成稳定 ID，前缀为类别首字母。）

**覆盖摘要表：**

| 需求键 | 是否有任务？ | 任务 ID | 备注 |
|--------|--------------|---------|------|

**宪章对齐问题：**（若有）

**未映射任务：**（若有）

**指标：**

- 需求总数
- 任务总数
- 覆盖率 %（至少关联 1 个任务的需求占比）
- 歧义数量
- 重复数量
- 严重问题数量

### 7. 提供后续动作

在报告末尾输出简洁「后续动作」块：

- 若存在 CRITICAL：建议在 `/speckit.implement` 之前先解决
- 若仅 LOW/MEDIUM：用户可继续，但给出改进建议
- 给出明确命令建议：例如「运行 /speckit.specify 细化」「运行 /speckit.plan 调整架构」「手动编辑 tasks.md 为『performance-metrics』补覆盖」

### 8. 提供修复选项

询问用户：「是否需要我为前 N 条问题建议具体修复编辑？」（**不要**自动应用。）

## 运行原则

### 上下文效率

- **最少高信号 token**：聚焦可执行发现，而非穷尽文档
- **渐进披露**：增量加载产物；勿将全文 dump 进分析
- **输出省 token**：发现表最多 50 行；溢出部分摘要
- **结果可复现**：无变更时重跑应得到一致的 ID 与计数

### 分析准则

- **绝不修改文件**（只读分析）
- **绝不臆造缺失章节**（若缺失，如实报告）
- **宪章违规优先**（恒为 CRITICAL）
- **用实例代替穷尽规则**（引用具体处，非泛泛模式）
- **零问题亦须优雅报告**（输出成功报告与覆盖统计）
