## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（若非空）。

## 概要

用户在触发消息中于 `/speckit.specify` 之后键入的文本**即**功能描述。即使下文出现字面量 `$ARGUMENTS`，仍假定你在本会话中始终可取得该描述。**不要**要求用户重复输入，除非命令为空。

在该功能描述基础上执行：

1. **生成分支简短名称**（2–4 个词）：
   - 分析功能描述并提取最有代表性的关键词
   - 生成 2–4 词的短名概括功能要点
   - 尽量使用「动作-名词」形式（如 `add-user-auth`、`fix-payment-bug`）
   - 保留技术术语与缩写（OAuth2、API、JWT 等）
   - 简洁但仍能一眼理解功能
   - 示例：
     - 「我要加用户认证」→ `user-auth`
     - 「为 API 实现 OAuth2 集成」→ `oauth2-api-integration`
     - 「做分析仪表盘」→ `analytics-dashboard`
     - 「修复支付处理超时 bug」→ `fix-payment-timeout`

2. **创建功能分支**：运行脚本时传入 `--short-name`（及 `--json`），**不要**传 `--number`（脚本会在所有分支与 spec 目录间自动检测下一个全局可用编号）：

   - Bash 示例：`.specify/scripts/bash/create-new-feature.sh "$ARGUMENTS" --json --short-name "user-auth" "Add user authentication"`
   - PowerShell 示例：`.specify/scripts/bash/create-new-feature.sh "$ARGUMENTS" -Json -ShortName "user-auth" "Add user authentication"`

   **重要**：
   - **不要**传 `--number` —— 脚本自动决定正确下一个编号
   - 始终包含 JSON 标志（Bash 用 `--json`，PowerShell 用 `-Json`）以便可靠解析输出
   - **每个功能**该脚本只能运行一次
   - JSON 在终端输出中提供——始终据此获取实际路径等内容
   - JSON 输出含 BRANCH_NAME 与 SPEC_FILE 路径
   - 参数中含单引号（如 "I'm Groot"）时使用转义：例如 `'I'\''m Groot'`（或尽量用双引号：`"I'm Groot"`）

3. 加载 `.specify/templates/spec-template.md` 以了解必填章节。

4. 按以下执行流：

    1. 从 Input 解析用户描述  
       若为空：ERROR "未提供功能描述"
    2. 从描述提取关键概念  
       识别：参与者、动作、数据、约束
    3. 对不清晰处：
       - 根据上下文与行业标准做合理推断
       - 仅在以下情况使用 `[NEEDS CLARIFICATION: specific question]` 标记：
         - 选择会显著影响功能范围或用户体验
         - 存在多种合理解读且含义不同
         - 不存在合理默认值
       - **上限：总计最多 3 处 `[NEEDS CLARIFICATION]`**
       - 按影响排序澄清：范围 > 安全/隐私 > 用户体验 > 技术细节
    4. 填写「用户场景与测试」节  
       若无清晰用户流：ERROR "无法确定用户场景"
    5. 生成功能需求  
       每条需求须可测试  
       对未写明细节使用合理默认值（在 Assumptions 节记录假设）
    6. 定义成功标准  
       创建可度量、与技术无关的结果  
       同时包含定量指标（时间、性能、体量）与定性指标（满意度、任务完成）  
       每条标准须可在不知实现细节的情况下验证
    7. 若涉及数据则识别关键实体
    8. 返回：SUCCESS（规格可进入计划阶段）

5. 使用模板结构将规格写入 SPEC_FILE，用从功能描述（参数）推导的具体内容替换占位符，保持章节顺序与标题不变。

6. **规格质量校验**：写入初稿后，按质量标准校验：

   a. **创建规格质量检查清单**：在 `FEATURE_DIR/checklists/requirements.md` 按检查清单模板结构生成，含下列校验项：

      ```markdown
      # 规格质量检查清单：[FEATURE NAME]
      
      **用途**：在进入计划前校验规格的完整性与质量
      **创建**：[DATE]
      **功能**：[链接至 spec.md]
      
      ## 内容质量
      
      - [ ] 无实现细节（语言、框架、API）
      - [ ] 聚焦用户价值与业务需求
      - [ ] 面向非技术干系人撰写
      - [ ] 所有必填章节已完成
      
      ## 需求完整性
      
      - [ ] 无残留的 [NEEDS CLARIFICATION] 标记
      - [ ] 需求可测试且无歧义
      - [ ] 成功标准可度量
      - [ ] 成功标准与技术无关（无实现细节）
      - [ ] 已定义全部验收场景
      - [ ] 已识别边界情况
      - [ ] 范围边界清晰
      - [ ] 已识别依赖与假设
      
      ## 功能就绪度
      
      - [ ] 所有功能需求均有明确验收标准
      - [ ] 用户场景覆盖主流程
      - [ ] 功能满足成功标准中定义的可度量结果
      - [ ] 规格中未渗入实现细节
      
      ## 说明
      
      - 未完成项须在运行 `/speckit.clarify` 或 `/speckit.plan` 前更新规格
      ```

   b. **执行校验**：逐项对照规格审查：
      - 每项判定通过或失败
      - 记录具体问题（引用相关规格段落）

   c. **处理校验结果**：

      - **若全部通过**：将检查清单标为完成并进入步骤 6（原步骤 7 报告中的步骤编号以用户文档为准——此处指「报告完成」前的逻辑）

      - **若存在失败项（不含未解决的 [NEEDS CLARIFICATION]）**：
        1. 列出失败项与具体问题
        2. 更新规格以逐项修复
        3. 重复校验直至全部通过（最多 3 轮）
        4. 若 3 轮后仍失败，在检查清单备注中记录剩余问题并警告用户

      - **若仍有 [NEEDS CLARIFICATION] 标记**：
        1. 从规格中提取所有 `[NEEDS CLARIFICATION: ...]` 标记
        2. **上限检查**：若超过 3 处，仅保留按范围/安全/UX 影响最大的 3 条，其余做合理推断
        3. 每条待澄清（最多 3 条）以下列格式向用户呈现选项：

           ```markdown
           ## 问题 [N]：[主题]
           
           **上下文**：[引用相关规格段落]
           
           **需明确**：[来自 NEEDS CLARIFICATION 标记的具体问题]
           
           **建议答案**：
           
           | 选项 | 答案 | 对功能的影响 |
           |------|------|----------------|
           | A      | [第一条建议答案] | [该选项对功能意味着什么] |
           | B      | [第二条建议答案] | [该选项对功能意味着什么] |
           | C      | [第三条建议答案] | [该选项对功能意味着什么] |
           | Custom | 自行填写答案 | [如何提供自定义输入] |
           
           **你的选择**：_[等待用户回复]_
           ```

        4. **关键 — 表格格式**：确保 Markdown 表格格式正确：
           - 管道符对齐、间距一致
           - 单元格内容两侧留空：`| Content |` 而非 `|Content|`
           - 表头分隔行至少三个连字符：`|--------|`
           - 在 Markdown 预览中确认表格可渲染
        5. 问题连续编号（Q1、Q2、Q3，总计最多 3 个）
        6. **在等待回复前一次性展示全部问题**
        7. 等待用户给出所有问题的选择（例如：`Q1: A, Q2: Custom - [details], Q3: B`）
        8. 用用户选定或提供的答案替换规格中各 `[NEEDS CLARIFICATION]` 标记
        9. 全部澄清解决后重新运行校验

   d. **更新检查清单**：每轮校验后更新检查清单文件的通过/失败状态

7. 报告完成：分支名、规格文件路径、检查清单结果、以及下一阶段就绪情况（`/speckit.clarify` 或 `/speckit.plan`）。

**说明：** 脚本会创建并检出新的功能分支，并在写入前初始化规格文件。

## 通用指南

## 快速指南

- 聚焦用户**需要什么**、**为什么**需要。
- 避免**如何实现**（技术栈、API、代码结构）。
- 面向业务干系人，而非开发者。
- **不要**在规格内嵌检查清单；检查清单由单独命令生成。

### 章节要求

- **必填章节**：每个功能都必须完成
- **可选章节**：仅在与功能相关时包含
- 某节不适用时**整节删除**（勿留「N/A」）

### AI 生成时

从用户提示创建规格时：

1. **合理推断**：用上下文、行业标准与常见模式填补空白
2. **记录假设**：在 Assumptions 节记录合理默认值
3. **限制澄清**：最多 3 处 `[NEEDS CLARIFICATION]` —— 仅用于以下关键决策：
   - 显著影响功能范围或用户体验
   - 多种合理解读且含义不同
   - 无任何合理默认值
4. **澄清优先级**：范围 > 安全/隐私 > 用户体验 > 技术细节
5. **测试者思维**：任何模糊需求都应无法通过「可测试且无歧义」检查项
6. **常见需澄清领域**（仅当无合理默认值时）：
   - 功能范围与边界（包含/排除哪些用例）
   - 用户类型与权限（若可能存在多种冲突解读）
   - 安全/合规要求（法律或财务意义重大时）

**合理默认值示例**（不必就此提问）：

- 数据保留：按领域常见实践
- 性能目标：除非另有说明，采用常规 Web/移动端预期
- 错误处理：用户可理解的提示与适当降级
- 认证方式：Web 应用常用会话或 OAuth2
- 集成模式：按项目类型选用合适模式（Web 服务用 REST/GraphQL，库用函数调用，工具用 CLI 参数等）

### 成功标准指南

成功标准必须：

1. **可度量**：含具体指标（时间、百分比、次数、比率）
2. **与技术无关**：不提框架、语言、数据库或工具
3. **用户导向**：从用户/业务视角描述结果，非系统内部
4. **可验证**：在不知实现细节的情况下仍可测试/验证

**良好示例**：

- 「用户可在 3 分钟内完成结账」
- 「系统支持 1 万并发用户」
- 「95% 的搜索在 1 秒内返回结果」
- 「任务完成率提升 40%」

**不佳示例**（实现导向）：

- 「API 响应时间低于 200ms」（过技术；可改为「用户即时看到结果」）
- 「数据库可支撑 1000 TPS」（实现细节；改用面向用户的指标）
- 「React 组件渲染高效」（框架相关）
- 「Redis 缓存命中率高于 80%」（技术特定）
