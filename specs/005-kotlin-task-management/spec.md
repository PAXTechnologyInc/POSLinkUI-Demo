## 澄清说明

### 会话 2025-03-19

- Q：当任务队列已满或执行器拒绝任务时，系统应如何处理？ → A：回退到调用方线程执行（在提交线程上执行任务以保证最终会被执行）
- Q：任务失败或取消时，用户何时应看到反馈（如 Toast、对话框），与仅打日志静默处理如何区分？ → A：仅对用户主动触发的取消给出可见反馈；由系统触发的取消与失败仅记录日志
- Q：同一 Fragment 或流程在已有超时任务运行时再次调度超时，应如何处理？ → A：替换——新任务取代旧任务，先前的超时任务被取消
- Q：未绑定具体界面（如打印检查、文件写入）的后台任务是否在本需求范围内？若在，生命周期如何管理？ → A：在范围内，显式取消——调用方须在适当时机显式取消；界面销毁时不会自动取消
- Q：任务取消应在何时完成——在 onDestroy 同步完成，还是 1 秒内异步完成？ → A：同步——取消必须在 onDestroy 期间完成，视为即时完成

## 用户场景与测试 *（必填）*

### 用户故事 1 - 生命周期感知的定时任务（优先级：P1）

当用户离开某界面或应用进入后台时，与该界面关联的已调度任务（超时、自动结束等）必须自动取消，以避免内存泄漏、组件销毁后回调仍触发导致的崩溃，以及状态不一致。

**优先级理由**：支付流程依赖超时与结束任务；若用户已离开仍触发，可能导致重复动作、错误 UI 更新或崩溃。

**独立测试**：启动超时任务后在触发前销毁 Activity/Fragment，验证无回调执行且无泄漏报告，即可完整验证。

**验收场景**：

1. **假如** 某界面存在进行中的超时任务，**当** 用户离开且组件被销毁，**那么** 任务在 onDestroy 期间被同步取消且无回调执行
2. **假如** 某界面存在进行中的结束任务，**当** Activity 被重建（如配置变更），**那么** 旧任务被取消且可为新实例重新调度
3. **假如** 多个 Fragment 各自请求超时任务，**当** 移除其中一个 Fragment，**那么** 仅该 Fragment 的任务被取消，其余继续

---

### 用户故事 2 - 主线程回调的后台 IO（优先级：P1）

当应用执行 IO 或计算（如打印检查、文件读取、网络准备）时，必须在非主线程运行并将结果安全投递到 UI；须支持取消，以便用户在完成前离开时工作可被中止。

**优先级理由**：阻塞主线程会导致 ANR；在已销毁视图上回调会导致崩溃；二者对支付可靠性均为关键。

**独立测试**：提交后台任务、在完成前取消，验证无回调执行且无崩溃。

**验收场景**：

1. **假如** 已提交后台任务，**当** 任务完成，**那么** 结果回调在主线程执行
2. **假如** 已提交后台任务，**当** 在完成前关联组件已销毁，**那么** 任务可被取消且无回调执行
3. **假如** 后台任务失败，**当** 失败发生，**那么** 错误被记录日志；按澄清约定，失败不向用户展示可见反馈（仅用户主动取消有可见反馈）

---

### 用户故事 3 - 倒计时与延迟 UI 更新（优先级：P2）

当界面展示倒计时（如签名超时）或需在延迟后更新 UI 时，延迟/倒计时必须与界面生命周期绑定；用户离开时倒计时停止，不再发生 UI 更新。

**优先级理由**：支付流程常见倒计时；过期更新易造成困惑与状态错误。

**独立测试**：启动倒计时后离开界面，验证倒计时停止且销毁后无 UI 更新。

**验收场景**：

1. **假如** 倒计时正在运行，**当** Fragment 被销毁，**那么** 倒计时被取消且不再发生后续 UI 更新
2. **假如** 已调度延迟 UI 更新，**当** 目标视图已无效，**那么** 跳过更新且无崩溃或泄漏

---

### 用户故事 4 - 统一的主线程投递（优先级：P2）

当代码需在主线程运行（如异步完成后，或推迟执行）时，须有统一、可测试的主线程投递方式；对已销毁目标的投递须安全（无崩溃、无泄漏）。

**优先级理由**：分散的 `Handler` 使用易出错（错误 Looper、无生命周期检查）；统一方式可提升可靠性与可测性。

**独立测试**：投递 Runnable，验证其在主线程执行；结合生命周期作用域投递，验证销毁后不再执行。

**验收场景**：

1. **假如** Runnable 被投递到主线程，**当** 投递执行时，**那么** Runnable 在主线程运行
2. **假如** 生命周期作用域内的投递，**当** 生命周期在投递执行前已销毁，**那么** 投递被跳过且无崩溃或泄漏

---

### 用户故事 5 - 结构化任务取消（优先级：P3）

当某流程（如一笔交易）涉及多个关联任务时，系统须支持一并取消，以避免用户取消或超时时的重复执行与状态不一致。

**优先级理由**：支付流程含多个协同任务；部分取消可能遗留孤儿任务继续运行。

**独立测试**：将任务分组后取消该组，验证全部任务被取消且无回调执行。

**验收场景**：

1. **假如** 一组任务（如超时 + 结束），**当** 取消该组，**那么** 组内全部任务被取消
2. **假如** 某任务，**当** 请求取消，**那么** 任务不再执行且无回调运行

---

### 边界情况

- **Activity 快速重建**：Activity 被快速销毁并重建（如旋转）时，须在调度新任务前取消旧任务
- **并发超时请求**：同一 Fragment 或流程在已有超时运行时再次调度，新任务必须替换旧任务（先前超时取消）；系统须避免重复执行
- **销毁后的主线程投递**：主线程投递已入队但执行前目标已销毁时，系统不得崩溃且不得泄漏引用
- **任务提交被拒**：队列满或执行器拒绝任务时，系统必须回退到在调用方线程执行以保证执行，且不得崩溃

## 需求 *（必填）*

### 功能需求

- **FR-001**：系统必须在关联的 Activity 或 Fragment 销毁时取消所有已调度任务（超时、结束、倒计时）；取消必须在 onDestroy 期间同步完成（视为即时）
- **FR-002**：系统必须提供统一的后台执行与主线程结果投递机制，并支持取消。任务可与生命周期绑定（自动取消）或由调用方管理（在适当时机显式取消）
- **FR-003**：系统必须支持可显式取消的延迟与周期性任务调度
- **FR-004**：系统必须提供统一的、生命周期感知的主线程工作投递方式
- **FR-005**：系统必须支持结构化取消（如取消一组关联任务），以避免重复执行与状态不一致
- **FR-006**：系统必须确保目标组件销毁后回调不再执行
- **FR-007**：系统必须记录任务取消与失败以供诊断，且不得记录敏感数据（遵循 payment-project-hard-problems）

### 关键实体

- **Scheduled Task（已调度任务）**：在未来某时刻或按固定间隔运行的任务；须可取消且可与生命周期绑定
- **Background Task（后台任务）**：在非主线程运行并通过主线程回调交付结果的任务；须支持取消。可与生命周期绑定（作用域销毁时自动取消）或由调用方管理（显式取消）
- **Task Group（任务组）**：可一并取消的逻辑任务分组（如某界面的超时 + 结束）
- **Lifecycle Scope（生命周期作用域）**：拥有任务的 Activity 或 Fragment 生命周期；作用域销毁时其任务被取消

## 成功标准 *（必填）*

### 可衡量成果

- **SC-001**：组件销毁时，其关联任务必须在 onDestroy 期间被同步取消，且后续无任何回调触发
- **SC-002**：后台任务失败或被取消时，系统必须记录该事件。可见反馈（如 Toast）仅针对用户主动取消；系统触发的取消与失败仅记录日志
- **SC-003**：在超时与取消场景下，交易状态机须到达明确终态（FAILED 或 CANCELLED），无歧义
- **SC-004**：任务调度不引入内存泄漏；组件销毁后可被垃圾回收
- **SC-005**：对已销毁目标的主线程投递不导致崩溃或泄漏

## 假设前提

- 项目已迁移至 Kotlin（参见 003-jetpack-refactor）；实现可采用 Kotlin 惯用法与现代并发原语
- 本规格技术中立；具体实现（协程、Executor 等）由技术方案（plan）决定
- 适用 payment-project-hard-problems 规则：状态机一致性、幂等与结构化日志
- 任务调度限于应用进程内；不需要分布式或跨进程调度
- 未绑定具体界面的后台任务（如打印检查、文件写入）在范围内；调用方必须在适当时机显式取消；界面销毁时不会自动取消
