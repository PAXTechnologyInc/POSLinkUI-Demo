# 合同：Entry Action 覆盖与验收契约

## 1. 覆盖完成定义

单个动作“完成”必须同时满足：

1. 可路由：动作触发后可进入正确交互页面；
2. 可提交：用户可完成输入/选择/确认并提交；
3. 可回写：提交结果可被后续交易流程读取。

任一条件不满足，该动作状态为“未完成”。

## 2. 验收输入/输出

### 输入

- 文档动作清单（首轮冻结基线 141/12 + 增量缺口清单）
- 动作分类信息（Text/Option/Confirmation/Information）

### 输出

- 每个动作的 `ActionCoverageItem` 结果
- 每个 category 的 `CategoryCoverageSummary`
- 人工验收记录与自动化回归清单

## 3. 失败处理契约

- Unknown action：记录为 `routable=false` 并阻断上线验收。
- 取消流程异常：若取消后流程悬挂，记为 `submittable=false`。
- 提交后无结果：若后续流程无法读取，记为 `writable=false`。
- 重复触发异常：同动作重复触发导致状态错乱，视为失败并阻断验收。

## 4. 自动化最小覆盖契约

- 本次涉及的每个 category 至少 1 个代表动作进入自动化回归。
- 自动化回归通过是 category 级完成门禁之一。

## 5. 安全与日志约束

- 验收日志中禁止输出 PAN、Track2、PIN、CVV 等敏感信息。
- 错误日志需包含动作标识与分类，便于排查，但不得包含敏感载荷明文。
