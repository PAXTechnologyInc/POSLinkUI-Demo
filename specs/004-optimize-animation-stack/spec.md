# 功能规格： 全量动画改造与性能降载

**特性分支**: `004-optimize-animation-stack`  
**创建日期**: 2026-03-19  
**状态**: 草案  
**输入**：用户描述： "项目中的动画库也需要改造，请选择一种合适的技术，从内存，从解决发烫，降低cpu的方向考虑，替换本项目中所有的动画"

## 动画实现方式对比（选型结论）

以下为**能力族**层面的差异，便于产品/研发对齐；具体类库与 API 映射见 [research.md](research.md)「库级差异」。


| 实现族                 | 典型用途                   | 相对内存    | 相对 CPU    | 跨机型/分辨率适配特点                                                               |
| ------------------- | ---------------------- | ------- | --------- | ------------------------------------------------------------------------- |
| **过渡类（声明式资源）**      | 页面/Fragment 进出、系统级淡入淡出 | 中等      | 中等        | 动画多按 **dp/百分比** 描述位移与缩放，一般不随物理像素变体单独出图；依赖布局约束保证容器尺寸合理即可                   |
| **视图属性类（运行时插值）**    | 单控件透明度、平移、缩放等          | 通常较低    | 通常较低      | 基于 **视图实际像素尺寸** 计算位移时，需在布局完成后测量或按比例换算，避免小屏裁切、大屏留白异常；**不要求**为每种分辨率重做切图     |
| **栅格序列类（位图/GIF 等）** | 边框动效、品牌审批动效            | 较高（帧缓存） | 较高（解码与绘制） | 易因固定像素资源导致糊或过大；**兼容策略**为：单套源资源 + **按目标区域限制解码/显示尺寸**（或矢量替代），避免「每档分辨率出一套新图」 |


**混合策略结论**：过渡保留声明式资源；高频闪烁/滑动用视图属性类降载；GIF 等用解码尺寸上限 + 降级策略控制内存与发热。

### 本项目涉及的「库/栈」对比（与实现族对应）

下表说明 **本仓库实际依赖与用法**（非全行业枚举）；API 级细节与迁移笔记见 [research.md](research.md) §2.3。


| 名称                                      | 类型          | 在本项目中的典型用途                           | 相对内存             | 相对 CPU         | 与同类相比的要点                                     |
| --------------------------------------- | ----------- | ------------------------------------ | ---------------- | -------------- | -------------------------------------------- |
| **Android Fragment + `res/anim` XML**   | 系统过渡        | A1/A2 页面与 Toast 进出                  | 中                | 中              | 与 `setCustomAnimations` 绑定；改 XML 即调曲线，无第三方依赖 |
| **ObjectAnimator / View.animate()**     | 系统属性动画      | A3 CLSS 指示灯闪烁/稳态降级                  | 低~中              | 低~中            | 比旧 `Animation`/`AnimationUtils` 更易与生命周期取消对齐  |
| **Coil + `coil-gif` + `coil-compose`**  | 图片加载（含 GIF） | A4 第二屏边框 GIF、A5 审批 GIF、A6 收据预览 crossfade | **高（取决于解码尺寸与帧）** | **高（解码+逐帧绘制）** | 统一图片栈后通过 Compose `AsyncImage` / `ComposeView` 复用 `ImageLoader`，减少双缓存与依赖体积 |
| **Jetpack Compose（非 `animate*`）**      | 声明式 UI 宿主    | 审批、收据预览与副屏整屏展示承载层               | 视实现              | 视实现            | Compose 负责页面承载，动图由 Coil 统一管理，避免在 View 层继续保留 `ImageView + load` |
| **Lottie（未引入）**                         | 矢量动效 JSON   | 当前 **未** 作为依赖使用                      | 通常低于大尺寸 GIF      | 中              | 若未来统一动效资产可考虑；需 **新资产形态**，与现有 GIF/XML 不是无痛替换  |


**当前结论**：项目已收敛为 **Coil 单图片栈**，并优先通过 Compose 承载层使用。对于 GIF 场景，优先通过 **限制解码尺寸 + 失败降级 + 共享 `ImageLoader`** 控制内存与发热，而不是继续保留双库或回到 `ImageView + load`。

### 各库/栈：采用原因与不采用替代方案的原因（汇总）

以下说明 **本需求范围内** 的取舍逻辑，便于评审与后续变更时对照；实现细节见 [research.md](research.md)。

| 库/栈 | **采用原因** | **未优先采用的替代** | **不采用替代的主要原因** |
|--------|--------------|----------------------|---------------------------|
| **Fragment + `res/anim` XML** | 已与 `setCustomAnimations` 深度耦合；过渡时长短，非内存/CPU 主矛盾；无额外依赖，易验收「时长/曲线与改造前一致」 | 全盘改为 AndroidX `Transition`、或过渡也改为 Lottie/Compose | 改动面大、回归成本高；对降热收益有限；与 FR-004a「视觉一致」冲突风险高 |
| **ObjectAnimator / View.animate()** | 属性动画管线与 `View` 生命周期对齐清晰，便于 `onDestroy`/detach 时取消；减少老式 `Animation` 无限循环带来的分配与回调开销 | 继续沿用 `AlphaAnimation` + `AnimationUtils`、或改为 Compose 动画 | 旧 API 在长时间闪烁场景下更不利 CPU/分配；Compose 要求页面 Compose 化，与 Entry 主链路现状不符 |
| **Coil + `coil-gif` + `coil-compose`** | A4/A5 动图与 A6 收据预览已统一到 Coil，并通过 Compose `AsyncImage` / 整屏 `ComposeView` 复用同一 `ImageLoader`；可用 `size()`、`crossfade(false)`、失败回退直接落策略 | 改回 Glide、或改为 Lottie | Glide 在当前代码已无主动调用，保留只会增加缓存与包体；Lottie 仍需要新资产形态，超出本需求边界 |
| **Jetpack Compose `animate*`** | 项目已引入 Compose，**新页或已 Compose 模块**可继续使用声明式动效 | 将 Entry 全链路动效统一迁到 Compose | Entry 以大量 Fragment 为主（参见 002-compose-migration）；全迁 Compose 是 **独立大专项**，不能与 004「动效降载」绑为同一交付单元 |
| **Lottie（当前未引入）** | — | 引入 Lottie 统一动效 | 现有资产为 **XML 过渡 + GIF**，上 Lottie 需 **设计侧产出/转换 JSON**；与 FR-004a、FR-009「尽量不强制重出图/换资产形态」冲突风险高；支付 UI 以短动效为主，**引入重型矢量管线性价比不足** |
| **单库图片栈（Coil）** | 当前主路径已覆盖第二屏、审批图与 Compose 图片场景；可减少双缓存、重复解码器与依赖维护 | 继续双库并存 | 双库并存的主要问题是 **包体、缓存与运维复杂度**，而继续保留 `ImageView + Coil` 也会让承载层分裂、策略分散 |

**总原则**：优先用 **约束解码、共享加载器、取消动画、策略档位、日志** 解决发热与内存；**换资产形态** 仅在设计统一或架构专项中立项，避免在单一「降载」需求内叠加不可逆范围。

## 用户场景与测试 *（必填）*

### 用户故事 1 - 交易主流程动画降载（优先级：P1）

作为支付终端操作员，我希望在交易主流程中看到更轻量且稳定的动画反馈，避免设备发热、卡顿和输入延迟。

**优先级理由**： 交易主流程是最高频路径，动画负担过高会直接影响交易完成率与用户体验。

**独立测试**： 仅执行交易主流程（启动交易、金额输入、确认、结果展示），即可验证动画替换后是否仍保持流程可用且性能改善。

**验收场景**：

1. **假如** 设备处于正常待机，**当** 连续完成 20 笔标准交易流程，**那么** 无因动画引起的页面卡顿或超过 200ms 的输入延迟（满足 SC-001）
2. **假如** 交易流程中存在页面切换与状态反馈，**当** 执行动画展示，**那么** 动画反馈完整且不影响关键按钮点击响应

---

### 用户故事 2 - 全量动画一致替换与降级（优先级：P1）

作为项目维护者，我希望项目内所有可控动画都统一替换到同一套性能优先标准，并对第三方不可控动画提供降级策略。

**优先级理由**： 若只替换部分动画，性能收益不稳定，且后续维护会出现多套动画行为并存问题。

**独立测试**： 对全项目动画清单逐项验收，验证每个动画位点已替换或已配置降级策略。

**验收场景**：

1. **假如** 项目动画资产与调用点清单，**当** 执行验收核对，**那么** 所有可控动画均已替换为性能优先标准
2. **假如** 第三方模块存在不可直接替换动画，**当** 运行应用，**那么** 相关动画已按预设策略降级或关闭并记录风险项

---

### 用户故事 3 - 长时运行温控稳定（优先级：P2）

作为门店终端管理员，我希望设备在长时间营业场景下保持温度与性能稳定，避免因动画负载导致掉帧或降频。

**优先级理由**： 终端常需长时运行，发热和 CPU 持续高占用会影响稳定性并增加故障率。

**独立测试**： 连续运行核心页面切换与交互场景 30 分钟，观测温度和性能指标是否满足阈值。

**验收场景**：

1. **假如** 连续运行场景已启动，**当** 运行 30 分钟，**那么** 设备表面温升与 CPU 占用满足 SC-002、SC-003、SC-004 阈值
2. **假如** 连续触发高频动画页面，**当** 进行操作，**那么** 页面仍保持可交互且 CPU 波动不超过基线 10%、内存峰值不超过基线 15%

---

### 用户故事 4 - 跨机型分辨率兼容（无需重新出图）（优先级：P2）

作为交付负责人，我希望在常见 POS 机型（不同物理分辨率与屏幕尺寸）上，动效与关键动效区域不出现不可接受的裁切、错位或模糊，且**不必**为适配每种机型单独重新导出全套切图。

**优先级理由**： 终端型号多，若依赖「每机型一套图」成本与版本管理风险高；通过约束与缩放策略可在不改设计主资产的前提下完成兼容。

**独立测试**： 在 **SC-007** 约定的机型/分辨率矩阵上执行同一套交易与动效路径，对照检查清单验收。

**验收场景**：

1. **假如** 矩阵内任一目标机型，**当** 执行含过渡、闪烁、小票滑动、第二屏边框、审批动效的路径，**那么** 关键控件可点、动效区域无遮挡交易主操作区
2. **假如** 使用现有设计交付的栅格类动效资源（如单套 GIF），**当** 在最小与最大目标分辨率设备上各跑一轮，**那么** 动效可读性可接受，且验收记录中**不要求**为该矩阵新增独立切图批次作为通过条件

---

### 边界情况

- 低性能设备在电量较低或系统后台任务较多时，动画仍需保持可交互性
- 前后台频繁切换时，不得出现动画状态错乱、黑屏或页面冻结
- 交易中断后恢复到流程页面时，动画状态应与业务状态一致，不出现重复播放干扰操作
- 第三方模块动画无法直接替换时，系统应自动启用降级策略并提示可追踪信息
- **小屏/大屏极端比例**：属性类动效的位移与缩放须基于安全区域或比例换算，避免控件移出可视区或 GIF 解码过大导致 OOM
- **第二屏与主屏分辨率不一致**：第二屏动效应独立按该屏容器适配，不强制与主屏共用同一像素尺寸资源

## 需求 *（必填）*

### 功能需求

- **FR-001**：系统必须建立项目级动画位点清单，覆盖页面切换、状态反馈、加载指示与提示动效
- **FR-002**：系统必须将所有可控动画替换为统一的性能优先动画标准，不得保留旧动画库并行路径
- **FR-003**：系统必须对不可控第三方动画提供统一降级策略（降频、简化或关闭）并支持按场景启停
- **FR-004**：系统必须保证动画替换后业务流程与 UI 功能语义不变，交易结果与交互结果保持一致
- **FR-004a**：系统必须保证动画迁移后视觉效果与改造前一致，不得改变原有动画的预览效果（时长、方向、缓动、可见元素）
- **FR-005**：系统必须在高频交互场景下保证关键操作响应可用，不得因动画导致关键操作超时或误触
- **FR-006**：系统必须建立动画改造后的性能验收流程，覆盖冷启动、主交易流程、长时运行、前后台切换
- **FR-006a**：系统必须提供可执行的自测用例，覆盖每个动画位点的视觉效果回归与性能基线对比，开发者可在本地一键运行
- **FR-007**：系统必须为每个高风险页面定义回退策略，确保性能异常时可切换到最简动画模式
- **FR-008**：系统必须记录动画降级触发与关键性能事件，支持定位发热和高负载来源
- **FR-009**：系统必须在目标交付的机型分辨率范围内完成动效兼容：**不得**将「为矩阵内每种分辨率单独重新导出全套切图」作为唯一适配手段；须采用可复用资产 + 布局约束、dp/比例描述、或按显示区域限制栅格类资源解码/显示尺寸等方式满足兼容

### 关键实体 *（若功能涉及数据则保留本节）*

- **AnimationInventoryItem**: 表示一个动画位点，包含页面上下文、触发时机、动画类型、是否可控、验收状态（当前主清单为 A1-A6）
- **AnimationPolicyProfile**: 表示动画策略档位，包含标准模式、降级模式、极简模式的触发条件与目标体验
- **PerformanceBaselineSnapshot**: 表示改造前后性能基线，包含 CPU、内存、温升、响应时延等验收指标
- **RiskExceptionRecord**: 表示不可控或未达标项，包含影响范围、临时策略、复核时间与责任归属
- **DeviceMatrixEntry**: 表示机型矩阵中的一条记录，含分辨率类别、主屏/副屏标识、验收勾选状态

## 成功标准 *（必填）*

### 可衡量成果

- **SC-001**: 主交易流程中，90% 以上页面交互在 200 毫秒内完成可点击响应
- **SC-002**: 与改造前基线相比，核心交易场景平均 CPU 占用下降至少 25%
- **SC-003**: 与改造前基线相比，核心交易场景内存峰值下降至少 20%
- **SC-004**: 连续运行 30 分钟后，设备温升较改造前基线降低至少 2 摄氏度
- **SC-005**: 全量动画位点中，可控位点替换完成率达到 100%，不可控位点均有生效的降级策略
- **SC-006**: 动画相关性能问题工单在上线后 2 个版本周期内下降至少 50%
- **SC-007**: 在书面约定的 **POS 机型矩阵**（至少覆盖「最小主流分辨率」「最大主流分辨率」各 1 台，且含第二屏场景时须含 1 台带第二屏设备）上，100% 通过动效与布局验收清单，且验收结论明确 **未新增** 因分辨率适配而要求的独立切图批次

## 范围外

- Neptune/POSLink SDK 接口变更
- 非动画相关的 UI 改造（由 002-compose-migration 负责）
- 新增业务功能或交易流程
- 动画设计稿或 Design Tokens 变更
- **主动改版视觉设计**（本需求下的兼容以「现有资产 + 工程适配」为主，不强制重做设计稿）

## 假设前提

- 当前业务要求为“功能与页面语义不变”，仅对动画实现方式和动画策略做改造
- 设备型号存在性能差异，因此验收以同批次设备对比基线为准
- 第三方 SDK 动画并非全部可替换，允许通过统一策略降级到可接受体验
- 项目已具备基本性能观测能力，可支撑 CPU、内存、温升和响应时延对比
- **机型矩阵**由项目/交付在 quickstart 或验收附件中书面列出；未列出前，SC-007 以团队默认最小集（至少 2 台分辨率档）为临时口径
