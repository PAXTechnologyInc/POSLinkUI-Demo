### 用户故事 1 - 连接可用性验证（优先级：P1）

作为开发者或运维，需要验证 SonarQube 服务器（http://172.16.3.60:9090）是否可达、是否正常运行，以便后续分析或规则同步可执行。

**优先级说明**：连接是后续所有操作的前提。

**独立测试**：调用 `/api/system/status`，验证返回 200 且 status 为 UP。

**验收场景**：

1. **假设** 网络正常，**当** 请求系统状态接口时，**则** 返回 HTTP 200 且包含版本与 UP 状态
2. **假设** 服务器不可达，**当** 请求超时或失败时，**则** 给出明确的连接失败提示

### 用户故事 2 - 获取 Sonar 规则（优先级：P2）

作为开发者或质量负责人，需要从 SonarQube 后台获取当前启用的规则列表（含规则键、名称、严重度、质量配置等），以便本地对齐规则、编写修复计划或导出文档。

**优先级说明**：规则是修复 Sonar 问题、统一质量标准的依据。

**独立测试**：在提供有效凭据的前提下，调用规则相关 API，验证能获取完整规则列表。

**验收场景**：

1. **假设** 已配置有效认证（Token 或用户名/密码），**当** 请求 `/api/rules/search` 或质量配置相关接口时，**则** 返回规则列表
2. **假设** 获取到规则，**当** 导出或展示时，**则** 至少包含规则键、名称、严重度、所属语言/配置
3. **假设** 未提供认证或认证无效，**当** 请求受保护接口时，**则** 返回明确错误（如 401）并提示需配置凭据

**优先级说明**：无集成则无法分析项目，是质量改进的基础。

**独立测试**：针对 SonarQube 服务器运行项目的分析配置，验证结果出现在看板中。

**验收场景**：

1. **假设** 项目构建成功，**当** 触发分析时，**则** 项目被成功分析，结果上传至 http://172.16.3.60:9090
2. **假设** SonarQube 服务器可达，**当** 分析完成时，**则** 项目出现在 SonarQube 项目列表中，且元数据正确

### 用户故事 4 - 解决全部 SonarQube 问题（优先级：P4）

作为维护者，SonarQube 针对本项目的所有报告问题必须被处理，使代码库满足部门质量门禁并降低技术债务。

**优先级说明**：这是提升代码质量与合规性的核心价值。

**独立测试**：运行 SonarQube 分析，验证质量门禁中无未解决的开问题（或仅有已接受/已确认的例外）。

### 用户故事 5 - Kotlin 属性访问与空安全（优先级：P5）

作为开发者，Kotlin 代码库必须遵循惯用风格（属性访问替代 getter 调用），并采用合理的 null 判断，使代码更易读、易维护。

**优先级说明**：提升可读性，降低 null 相关 bug 风险；与 SonarQube 修复互补。

**独立测试**：代码审查和静态分析确认属性访问语法与 null 判断符合 Kotlin 规范。

**验收场景**：

1. **假设** 存在调用 Java 风格 getter 的代码（如 `obj.getProperty()`），**当** 对象为 Kotlin 或支持 Kotlin 属性访问时，**则** 转换为属性语法（`obj.property`）
2. **假设** 存在 null 判断代码，**当** 审查时，**则** 在合适处使用安全调用（`?.`）、Elvis 运算符（`?:`）或 `let`/`also`
3. **假设** 已转换代码，**当** 项目构建并运行测试时，**则** 行为保持不变

## 澄清说明

### 会话 2025-03-20（原 007）

- Q: 凭据应如何提供给获取规则的脚本/工具？ → A: 环境变量优先，可选本地配置文件（如 sonar-project.properties 中的 sonar.token）。注：SonarQube MCP 已配置 Token，通过 MCP 获取规则时无需额外凭据。
- Q: 获取规则时，应以哪种范围为准？ → A: 全服务器所有规则（不区分项目/配置）。
- Q: 当规则数量很大时，应采用哪种获取方式？ → A: 自动分页拉取并合并为完整列表。
- Q: 连接超时或请求失败时，应如何处理？ → A: 简单重试（如最多 2 次，间隔 3–5 秒）。
- Q: 规则列表的导出/输出应以哪种格式为主？ → A: JSON（主格式，便于程序消费）。
- Q: getter→属性访问的收益是什么？ → A: 可缩减代码语句、提升可读性。例如 `obj.getX()`→`obj.x` 少 6 字符；链式 `a.getB().getC()`→`a.b.c` 更简洁；条件 `obj?.getX()?.getY()`→`obj?.x?.y` 更符合 Kotlin 惯用法。Android API 如 `activity.getSupportFragmentManager().executePendingTransactions()`→`activity.supportFragmentManager.executePendingTransactions()` 同理。
- Q: 多分支 if-else 如何改进？ → A: 将同一变量的多分支判断替换为 `when` 表达式，如 `if (x == A) ... else if (x == B) ...` → `when (x) { A -> ... B -> ... else -> ... }`，更简洁且利于穷举；适用 StatusFragment、UIFragmentHelper、TransactionPresentation 等含状态/类型分支的类。

## 假设前提

- SonarQube 服务器 http://172.16.3.60:9090 在开发/CI 网络内可达
- 用户可获取 SonarQube Token 或用户名/密码（由管理员分配）；若使用 MCP，则 MCP 已配置 Token
- 规则 API 路径遵循 SonarQube 标准（如 `/api/rules/search`、`/api/qualityprofiles/search`）
- 获取规则的目的为本地对齐、修复计划或文档导出，不要求实时同步；输出以 JSON 为主格式
- 服务器上的质量门禁定义可接受；本功能无需修改门禁规则
- 「Use of getter method instead of property access syntax」对应 SonarQube 规则 S6518 或等效规则
- null 判断改进聚焦 Kotlin 空安全构造；除非必要，不修改 Java 互操作契约
